# AI ネイティブ運用 OS

このリポジトリは自分の仕事を AI ネイティブに再設計するための「運用 OS」。
Claude Code / Claude アプリが常時参照できる形で、コンテキスト・テンプレ・プロンプト・手順書を一元管理する。

## 原則

1. **コンテキストイズキング**：質の良いコンテキストが無ければ AI は鈍る。情報は Markdown で資産化する。
2. **短く投げる**：プロンプトは一〜二文が基本。足りない情報は AI に逆質問させる。
3. **毎日追記**：発見・やり取り・顧客情報は 30 秒で該当ファイルに追記。音声入力で吐いて OK。
4. **委任率を上げる**：手で書く業務文章は週ごとに減らす。目標は 10% 以下。
5. **秘密情報の境界**：PII・契約・パスワードは直接書かず、`[REDACTED]` 等で伏せる。

## ディレクトリ

```
.
├── CLAUDE.md              # このファイル（運用ルール）
├── me.md                  # 自分の取扱説明書
├── context/
│   ├── clients/           # 顧客別ファイル（<name>.md）
│   └── projects/          # 案件別ファイル（<name>.md）
├── templates/             # 書類の Markdown 雛形
├── prompts/               # 一〜二文の定型プロンプト
├── playbooks/             # 業務フロー手順書
└── scripts/               # 自動化スクリプト（Claude Code に書かせる）
```

## Claude Code への指示

- 書類生成を依頼されたら、`templates/` の雛形をベースに `context/` と `me.md` を参照して埋める。
- 顧客情報は `context/clients/<name>.md` を正として扱う。矛盾があれば人間に確認する。
- 出力は Markdown。必要なら PDF/Word は `scripts/` 側で変換する。
- 生成後、使用した context/template のパスを最後に列挙する（再現性のため）。
- 足りないコンテキストは勝手に補完せず、**不足項目を箇条書きで返す**。

## 典型フロー

1. 「`clients/acme.md` と `projects/q2-launch.md` から 4 月分請求書作って」
2. Claude Code が `templates/invoice.md` を読み、`context/` から情報を埋める
3. 不足があれば質問、揃っていれば `outputs/` または標準出力に生成
4. 内容確認 → 必要箇所を人間が手直し → 送付

## ツール連携（MCP / API）

竹村が日常使うツールは全て AI と連携する方針。
Claude Code は MCP サーバー経由で外部ツールに接続できる。

| ツール | 用途 | 連携状態 |
|--------|------|----------|
| **サイボウズ** | メール・社内グループウェア | 未接続 |
| **Asana** | プロジェクト・タスク・組織マネジメント | 未接続 |
| **Google カレンダー** | スケジュール管理 | 未接続 |
| **GitHub** | この運用 OS リポジトリ | 接続済 |

### 連携時の原則
- 接続したツールの情報はコンテキストとして活用してよい
- ツール間の情報を突合して矛盾があれば人間に確認する
- 仕入価格など `me.md` の NG 事項に該当する情報は、ツール経由で取得しても出力しない
- 新しいツールを接続したらこの表に追記する

## 更新のしかた

- 顧客と話した直後：`context/clients/<name>.md` の末尾に日付付きで追記
- 案件の進捗変化：`context/projects/<name>.md` の状態を更新
- 新しい定型業務を見つけたら：`prompts/` に一〜二文で追加、必要なら `playbooks/` に手順化
- 自分の判断基準・文体が変わったら：`me.md` を更新
